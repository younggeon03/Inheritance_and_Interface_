# Java 다형성과 설계 (Inheritance & Interface)

이 저장소는 자바의 핵심 원칙인 **다형성(Polymorphism)**을 이해하고, 이를 실무적인 설계(인터페이스, 추상 클래스)에 적용하는 과정을 학습한 기록입니다.

## 1. 학습 목표: 좋은 객체 지향 프로그래밍이란?
* 프로그램을 단순한 명령어 목록이 아닌 독립된 단위인 **'객체'들의 모임**으로 파악합니다.
* 객체 간의 **협력** 관계를 통해 프로그램을 유연하고 변경이 용이하게 만드는 방법을 익힙니다.
* **역할(Role)**과 **구현(Implementation)**을 분리하여 확장 가능한 설계를 지향합니다.

---

## 2. 핵심 설계 원칙: 역할과 구현의 분리
실세계의 역할과 구현이라는 컨셉을 객체 지향의 다형성으로 구현합니다.

### 🛠️ 역할(Role) = 인터페이스(Interface)
* 클라이언트는 대상의 **역할(인터페이스)**만 알면 됩니다.
* 구현 대상의 내부 구조를 몰라도 되며, 구조가 변경되거나 구현 객체 자체가 바뀌어도 영향을 받지 않습니다.
* 객체 설계 시 역할(인터페이스)을 먼저 부여하고, 그 역할을 수행하는 구현 객체를 나중에 만듭니다.

### 🧬 구현(Implementation) = 구현 클래스(Implementation Class)
* 인터페이스를 상속받아 구체적인 기능을 수행하는 실제 객체입니다.
* 실행 시점에 구현 객체를 유연하게 변경할 수 있는 것이 **다형성의 본질**입니다.

---

## 3. 좋은 객체 지향 설계 원칙 (OCP)
가장 중요한 설계 원칙 중 하나인 **OCP(Open-Closed Principle)**를 코드로 실천합니다.

* **Open for extension**: 새로운 기능의 추가나 변경 사항이 생겼을 때, 기존 코드는 확장할 수 있어야 합니다.
* **Closed for modification**: 새로운 기능을 추가해도 기존의 핵심 클라이언트 코드(예: Driver, PayService)는 수정되지 않아야 합니다.
* **효과**: 다형성을 활용하여 핵심 로직의 수정 없이 서버의 구현 기능을 유연하게 변경할 수 있습니다.

---

## 4. 적용 프로젝트 및 디자인 패턴

### 🏎️ 자동차 운전자 예제
* **역할**: `Car` 인터페이스
* **구현**: `K3Car`, `Model3Car`
* **특징**: 운전자(`Driver`)는 특정 차종이 아닌 `Car` 역할에만 의존하여, 새로운 차종이 추가되어도 운전자 코드를 변경하지 않습니다.

### 💳 결제 시스템 리팩토링 (전략 패턴)
* **전략 패턴(Strategy Pattern)**: 알고리즘(결제 방식)을 클라이언트 코드의 변경 없이 쉽게 교체할 수 있게 설계했습니다.
* **구조**:
    * `Pay` 인터페이스를 통한 결제 전략 정의
    * `PayStore` 추상 클래스를 활용하여 객체 생성을 막고 정적 메서드로 결제 수단 조회 로직 은닉
    * `DefaultPay`를 사용하여 `null` 체크 없이 결제 수단 부재 상황을 안전하게 처리

## 5. 요약 및 시사점
* **다형성**은 객체 지향 설계의 꽃이며, 디자인 패턴의 대부분은 다형성을 활용합니다.
* 스프링(Spring)의 핵심인 **제어의 역전(IoC)**과 **의존관계 주입(DI)** 역시 결국 다형성을 극대화한 결과물입니다.
* 인터페이스를 안정적으로 잘 설계하는 것이 확장에 유리한 프로그램을 만드는 지름길입니다.
