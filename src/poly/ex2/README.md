# 다형성과 설계 (Polymorphism & Design)

## 1. 객체 지향 프로그래밍의 본질
* **객체 중심의 설계**: 프로그램을 단순한 명령어 목록이 아닌, 독립된 단위인 '객체'들의 모임으로 파악합니다.
* **협력과 메시지**: 각각의 객체는 서로 메시지를 주고받으며 데이터를 처리하고 협력합니다.
* **유연성과 변경 용이성**: 레고 블록을 조립하듯이, 컴포넌트를 쉽고 유연하게 변경할 수 있는 설계를 지향합니다.

---

## 2. 역할과 구현의 분리 (Interface vs Class)
세상을 **역할**과 **구현**으로 구분하면 설계가 단순해지고 확장이 편리해집니다.



### 🛠️ 인터페이스 (역할)
* **표준 규격**: 자바 언어에서 '역할'은 인터페이스를 통해 정의합니다.
* **클라이언트 독립성**: 클라이언트는 대상의 역할(인터페이스)만 알면 되며, 내부 구조를 몰라도 영향을 받지 않습니다.
* **다형적 활용**: 실행 시점에 구현 객체를 유연하게 변경하여 시스템의 유연성을 극대화합니다.

### 🧬 추상 클래스 (상태와 제약)
* **인스턴스 생성 방지**: 객체 생성을 막아야 하는 유틸리티성 클래스(예: `PayStore`) 설계 시 사용합니다.
* **상속 기반 다형성**: 클래스 상속 관계에서도 오버라이딩을 통한 다형성을 적용할 수 있습니다.

---

## 3. 좋은 객체 지향 설계 원칙 (OCP)
기존 코드를 수정하지 않고 기능을 확장하는 **OCP(Open-Closed Principle)** 원칙을 준수합니다.



* **Open for extension**: 새로운 기능(차량 추가, 결제 수단 추가 등)에 대해 열려 있어 확장이 자유롭습니다.
* **Closed for modification**: 기능을 확장해도 기존의 핵심 클라이언트 코드(`Driver`, `PayService`)는 수정되지 않아야 합니다.
* **핵심 가치**: 다형성을 활용하여 로직 수정 없이 구현 기능을 유연하게 변경하는 것입니다.

---

## 4. 실전 적용: 결제 시스템 (Strategy Pattern)
* **전략 패턴(Strategy Pattern)**: 결제 방식이라는 '전략'을 인터페이스로 정의하고, 클라이언트 코드 변경 없이 전략을 자유롭게 교체합니다.
* **반환 타입 (boolean)**: 결제 성공 여부를 응답하여 클라이언트가 결과에 따라 비즈니스 로직을 제어할 수 있게 설계했습니다.
* **Null Object 패턴 (DefaultPay)**: 결제 수단을 찾지 못했을 때 `null` 대신 실패 전용 객체를 반환하여, `null` 체크 로직을 제거하고 코드의 안정성을 높였습니다.

---

## 5. 설계 시 유의점 및 한계
* **안정적 설계**: 인터페이스(역할) 자체가 변하면 클라이언트와 서버 모두에 큰 변경이 발생하므로 초기 설계가 매우 중요합니다.
* **확장성**: 다형성을 잘 활용한 설계는 스프링(Spring)의 핵심인 제어의 역전(IoC)과 의존관계 주입(DI)의 기반이 됩니다.
